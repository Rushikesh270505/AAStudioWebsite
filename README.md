# Art and Architecture Studios Platform

A premium architecture studio website plus SaaS delivery layer built with Next.js, React Three Fiber, Express, and MongoDB.

## What is included

- Public portfolio website
- Project showcase pages
- Real-time 3D viewer
- Browser-based walkthrough engine
- Client dashboard
- Architect admin panel
- JWT authentication
- Role-based API access
- MongoDB models for users, projects, files, messages, and invoices
- Cloud storage abstraction for local, AWS S3, or Cloudinary uploads

## Stack

- Frontend: Next.js 16, React 19, Tailwind CSS 4, Framer Motion, React Three Fiber, Drei
- Backend: Express, Node.js
- Database: MongoDB with Mongoose
- Storage: local dev storage with switchable S3 / Cloudinary providers
- Auth: JWT with roles for architect, client, and admin

## Local setup

1. Install dependencies:

   ```bash
   npm install
   npm --prefix frontend install
   npm --prefix backend install
   ```

2. Create env files:

   ```bash
   cp frontend/.env.example frontend/.env.local
   cp backend/.env.example backend/.env
   ```

3. Start MongoDB locally, or update `backend/.env` with your hosted MongoDB URI.

4. Seed demo data:

   Optionally set `SEED_ARCHITECT_PASSWORD`, `SEED_CLIENT_PASSWORD`, and `SEED_ADMIN_PASSWORD` in `backend/.env` first if you want stable demo passwords. If you leave them blank, the seed script will generate strong random passwords and print them in the terminal.

   ```bash
   npm run seed
   ```

5. Run both apps:

   ```bash
   npm run dev
   ```

6. Open:
- Frontend: `http://localhost:3000`
- Backend: `http://localhost:5001`
- Health check: `http://localhost:5001/api/health`

## Seeded demo users

- Architect: `architect.demo@example.com`
- Client: `client.demo@example.com`
- Admin: `admin.demo@example.com`

The passwords are supplied via `backend/.env` at seed time or generated on the fly by `npm run seed`. They are intentionally not stored in the repository.

## Storage configuration

`backend/.env` supports:

- `STORAGE_PROVIDER=local`
- `STORAGE_PROVIDER=s3`
- `STORAGE_PROVIDER=cloudinary`

For `s3`, set:

- `AWS_REGION`
- `AWS_ACCESS_KEY_ID`
- `AWS_SECRET_ACCESS_KEY`
- `AWS_S3_BUCKET`

For `cloudinary`, set:

- `CLOUDINARY_CLOUD_NAME`
- `CLOUDINARY_API_KEY`
- `CLOUDINARY_API_SECRET`

## Deployment

### Frontend on Vercel

This repo is set up as a monorepo deployment. Create a Vercel project from this GitHub repo and set the root directory to `frontend`.

Recommended Vercel settings:

- Framework preset: `Next.js`
- Root directory: `frontend`
- Production branch: `main`
- Node.js version: `22.x`

Required environment variables in Vercel:

- `NEXT_PUBLIC_API_URL=https://<your-render-service>.onrender.com/api`

Vercel setup steps:

1. Import the GitHub repo into Vercel.
2. In the import flow, set the Root Directory to `frontend`.
3. In Project Settings, set Node.js Version to `22.x` if Vercel does not infer it from `frontend/package.json`.
4. Add `NEXT_PUBLIC_API_URL` for Production and Preview environments.
5. Deploy.

### Backend on Render

This repo includes a Blueprint file at [`render.yaml`](./render.yaml) for the backend service.

Recommended Render settings:

- Service type: `Web Service`
- Root directory: `backend`
- Runtime: `Node`
- Plan in Blueprint: `free` by default
- Health check path: `/api/health`
- Node version: `22.22.0`

Required environment variables in Render:

- `MONGODB_URI`
- `CLIENT_URL`
- `SERVER_URL`
- `JWT_SECRET` is auto-generated by the Blueprint

Optional but recommended for production file uploads:

- Set `STORAGE_PROVIDER=cloudinary` and configure:
  - `CLOUDINARY_CLOUD_NAME`
  - `CLOUDINARY_API_KEY`
  - `CLOUDINARY_API_SECRET`

Render setup steps:

1. In Render, choose `New > Blueprint`.
2. Connect the GitHub repo.
3. Select the `main` branch.
4. Review the generated `aa-architecture-api` service from `render.yaml`.
5. Fill in `MONGODB_URI`, `CLIENT_URL`, and `SERVER_URL` before the first deploy.
6. Sync the Blueprint.

Suggested Render environment values:

- `CLIENT_URL=https://<your-vercel-project>.vercel.app,https://*.vercel.app`
- `SERVER_URL=https://<your-render-service>.onrender.com`

### Deployment order

1. Deploy the backend to Render first and copy the final `.onrender.com` URL.
2. Add that URL as `NEXT_PUBLIC_API_URL` in Vercel.
3. Deploy the frontend to Vercel.
4. Update Render `CLIENT_URL` with the final Vercel domain.
5. Redeploy the backend if you changed Render environment variables.

### MongoDB for production

Render does not provision MongoDB in this repo because the application uses MongoDB, not Postgres. Use MongoDB Atlas or another hosted MongoDB provider and set the connection string as `MONGODB_URI`.

### Notes for previews

- The backend CORS configuration accepts comma-separated origins and wildcard Vercel subdomains such as `https://*.vercel.app`.
- If you use Vercel Preview deployments, keep `NEXT_PUBLIC_API_URL` configured for the Render backend in the Preview environment as well.
- Local filesystem uploads are acceptable for development only. Render disks are not configured here, so use Cloudinary or S3 for production assets.

## Notes

- The current brand mark is a clean AA monogram inspired by your logo reference. If you want the exact logo asset used, place it in `frontend/public/brand/` and swap the `BrandMark` component.
- The 3D viewer is production-ready for GLB / GLTF loading. The demo falls back to a parametric architecture model until real studio assets are uploaded.
- The walkthrough scene is implemented as a configurable engine scaffold and can be replaced with project-specific geometry or imported GLB interiors.
